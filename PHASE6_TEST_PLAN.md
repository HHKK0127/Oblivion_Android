# Phase 6 実デバイステスト・最適化計画

## テスト概要

実デバイスでプロファイリングデータを収集し、パフォーマンスボトルネックを特定して最適化する。

**目標**: 最小スペックデバイス（G001MG06315404FA）で 30 fps 以上を安定保持

---

## テスト環境

### テストデバイス

1. **G001MG06315404FA** (メインテスト対象)
   - API Level: 29+ (Android 10)
   - メモリ: 不明（推定 2-4GB）
   - CPU: 不明（推定 Snapdragon 400-600 系）
   - ステータス: `adb devices` で確認可能

2. **192.168.3.17:5555** (ネットワーク接続デバイス)
   - 補助テスト対象
   - ワイヤレス接続

### テスト項目

| 項目 | 目標値 | 現状 | 対策 |
|-----|------|------|-----|
| FPS (30fps設定) | 30 fps 安定 | 計測予定 | ボトルネック最適化 |
| FPS (60fps設定) | 50-60 fps | 計測予定 | 不要な計算削減 |
| メモリ使用率 | < 60% | 計測予定 | キャッシュ削減、アセット圧縮 |
| CPU使用率 | < 80% | 計測予定 | AI計算削減、物理演算簡略化 |
| バッテリー (1時間) | > 30% 残量 | 計測予定 | 消費電力削減 |

## テスト手順

### Step 1: APK ビルド & デバイスへのインストール

```bash
# プロジェクトディレクトリで実行
cd I:\Closet\Oblivion\Projects\oblivion-android

# ビルド

# または Android Studio から Build → Build APK

# APK パス: app/build/outputs/apk/debug/app-debug.apk
```

**期待される出力**:
```

```

### Step 2: デバイスへのインストール

```bash
# デバイスをUSB接続確認
adb -s G001MG06315404FA devices

# APK をインストール
adb -s G001MG06315404FA install -r app/build/outputs/apk/debug/app-debug.apk

# または
adb install -r app/build/outputs/apk/debug/app-debug.apk
```

**期待される出力**:
```
Success
```

### Step 3: アプリ起動 & ログ収集

```bash
# Logcat をリアルタイム監視
adb -s G001MG06315404FA logcat | grep -E "Renderer|PerformanceMonitor|FPS"

# または、ログファイルに保存
adb -s G001MG06315404FA logcat > profiling_log.txt &

# アプリを起動（デバイス上で手動操作）
# または
adb shell am start -n com.example.oblivion/.MainActivity
```

### Step 4: ゲーム実行 & データ収集

**実行シーン**:

1. **タイトル画面** (3秒)
   - ロゴ表示フェーズ
   - 計測内容: FPS、メモリ使用率

2. **ゲーム開始** (5分)
   - メインゲーム実行
   - NPC 移動・戦闘
   - 魔法キャスト
   - 計測内容: 全メトリクス

3. **FPS 変更テスト** (各30秒 × 3)
   ```
   30 FPS 設定 → 30秒実行
   60 FPS 設定 → 30秒実行
   120 FPS 設定 → 30秒実行（デバイスが対応していれば）
   ```

4. **メモリストレステスト** (2-3分)
   - 同時 NPC を増やして実行
   - メモリ使用率の推移を観察

5. **バッテリー消費テスト**
   - 1時間連続プレイ
   - バッテリー消費量を記録

## 予想ボトルネック & 対策

### パターン1: Render Time が高い（> 10ms @60fps）

**原因**:
- 描画バッチ数が多い
- テクスチャ切り替えが頻繁
- シェーダーが複雑

**対策**:
```cpp
// 1. バッチ処理の最適化
// 同じマテリアルのメッシュをまとめて描画

// 2. バッチ削減
// draw call の統合

// 3. Culling の改善
// 視錐台外のオブジェクトをスキップ
```

### パターン2: Update Time が高い（> 5ms @60fps）

**原因**:
- NPC AI 計算が毎フレーム実行
- 物理演算が重い
- 衝突判定が多い

**対策**:
```cpp
// 1. AI 計算頻度削減
// 5フレーム毎に AI 更新

// 2. 物理演算簡略化
// Bullet Physics の精度を低下

// 3. 衝突判定最適化
// Spatial hash grid 導入
```

### パターン3: メモリ使用率が増加（> 70%）

**原因**:
- アセットキャッシュが満杯
- NPC インスタンスが多すぎる
- テクスチャが圧縮されていない

**対策**:
```cpp
// 1. LRU キャッシュ実装
// 古いアセットを削除

// 2. NPC pooling
// 最大 50 NPC に制限

// 3. テクスチャ圧縮
// ETC2/ASTC 形式に変換
```

### パターン4: CPU使用率が高い（> 80%）

**原因**:
- スレッド数が多い
- メモリアロケーションが頻繁
- 文字列操作が多い

**対策**:
```cpp
// 1. メモリプール実装
// new/delete 削減

// 2. 文字列キャッシング
// 動的生成を事前計算

// 3. スレッド最適化
// 不要なスレッド削除
```

## データ解析方法

### 1. Logcat からのデータ抽出

```bash
# 5秒毎のパフォーマンスレポートを抽出
grep "Performance Report" profiling_log.txt -A 5

# 詳細メトリクスを抽出
grep "Detailed Metrics" profiling_log.txt -A 20
```

### 2. Excel / スプレッドシートへの入力

収集したデータを以下の形式で記録：

| 時刻 | FPS設定 | 実FPS | 平均フレーム時間(ms) | CPU使用率(%) | メモリ使用率(%) | ドロップフレーム |
|-----|--------|------|----------------|-----------|------------|--------|
| 10:00 | 60 | 59.8 | 16.7 | 45 | 52 | 0 |
| 10:05 | 60 | 58.5 | 17.1 | 48 | 54 | 2 |
| 10:10 | 30 | 30.0 | 33.3 | 22 | 48 | 0 |

### 3. グラフ化

- FPS の時系列変化
- メモリ使用率の推移
- CPU使用率の推移
- ドロップフレーム数
